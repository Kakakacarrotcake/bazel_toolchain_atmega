1.   Overview

1.1.     Introduction

This release of the Microchip Hexmate utility is the first as a stand-alone product. The changes are mostly internal and relate to the build process, but this release does additionally contain several new features, improvements, and bug fixes.

1.2.     Build Date

The official build date of this version is the 26 October 2023.

1.3.     Previous Version

The previous Hexmate version was 2.42, built on 28 March, 2023.

1.4.     System Requirements

The Hexmate utility us available for a variety of operating systems, including 64-bit versions of the following: Professional editions of Microsoft® Windows® 10, Ubuntu® 18.04, macOS® 13.2 (Ventura), and Fedora 34. Binaries for Windows have been code-signed. Binaries for macOS have been code-signed and notarized.

1.5.     Devices Supported

Hexmate operates on and generates files conforming to the Hexadecimal Object File Format developed by Intel®. All Microchip MPLAB XC compilers can generate these files, regardless of target device. Thus, Hexmate is usable with all devices. Note, however, that Hexmate acts only on HEX files. It cannot process ELF, COFF, BIN or any other format.

1.6.     Editions and License Upgrades

Hexmate does not require a license to operate, nor is any license available for purchase. All Hexmate features are available to all users.

1.7.     Installation

Hexmate is automatically installed with the MPLAB XC8 compiler. It is also installed with the MPLAB X IDE. You should not typically need to install Hexmate yourself; however, there are no special installation steps required nor any external dependencies to prevent you from doing so. The application can be copied to any desired location that supports normal execution.

1.8.     Hexmate Documentation

The Hexmate User's Guide (DS50003033) describes the options and usage of the Hexmate application. It is available in the docs directory of your Hexmate distribution.

If you are controlling Hexmate features from the MPLAB XC8 compiler, then there is a chapter the in the MPLAB® XC8 C Compiler User’s Guide for PIC® MCU (DS50002737) which contains similar information.

1.9.     Customer Support

You can ask questions of other users of this product in the XC8 Forum.

Microchip welcomes bug reports, suggestions or comments regarding this compiler version. Please direct any bug reports or feature requests via the Support System.

3.   What's New

3.1.     Version 3.00

New SHA1 hash algorithm (HEXMATE-14) The 160-bit SHA1 algorithm has been added as a selection when using the -CK option to generate a hash value. Use algorithm number 11 with the g suboption. The width must be specified as ±160, for example, -ck=700-705@100w160g11s2.

Increased hash width (HEXMATE-13) The allowable hash width of all additive, subtractive, and CRC hash algorithms (selectors -5 through -1 and 1 through 5) has been increased to 8 bytes. As has always been the case, specify the width you require with the w suboption in the -CK option.

New warning option (HEXMATE-11) A new option, --werror, which can promote warnings to errors, has been added. Without an argument, the option affects all warnings, but individual warnings, identified by their number, can be specified as an argument. The --wno-error form of this option can be used to prevent warnings being promoted to errors when all warnings have been promoted.

3.2.     Version 2.45

Mac universal binaries (XC8-3168, XC8-3116, XC8-2951, XC8-2693) The compiler binary files for macOS are now universal, offering native support for both Intel- and M1-based machines.

Hexmate comment support (XC8-3144) Hexmate now supports the presence of comments in HEX files. A comment is considered to be any line containing a sequence of characters that does not include a colon character, any sequence of characters before a colon character that starts a valid record, or any sequence of characters that includes a colon character that is not followed by a valid record. Although comments are permitted in input files, they are not generated in any output.

3.8.     Version 2.35

Wider use of reverse-word hashes The Hexmate reverse-word processing hash calculation feature can now be applied to Fletcher algorithms. This feature was previously applicable to only CRC and SHA algorithms. It has no affect on the additive or subtractive algorithms.

Final XOR with Hexmate hashes Hexmate can now be asked to perform an XOR of a hash result with a specified value. Use o or O followed by the value as an additional argument to Hexmate's -CK option.

I16HEX file support in Hexmate Hexmate has been extended to read type 2 records, as well as a new --ssa option to specify the Start Segment Address used by type 3 records. INHX16 files can be written by using the new inhx16 argument to the --format option.

Hexmate II The Hexmate application shipped with the compiler is now built from a new code base. Although the application's features and the options that drive it are mostly identical to those in the previous incarnation, the parsing of its options is much improved. See also Migration Issues.

3.11.  Version 2.30

Hexmate hash calculations Hexmate has two new hash-calculation features. Bytes in the HEX file can be skipped for the purposes of calculating a hash value by using a new 's' argument to Hexmate's -CK option. This would be useful when there is data in the Hex file that is not present on the device, such as padding bytes added by the compiler. Another extension to the -CK option's 't' argument allows the trailing code sequence to be appended to a specified number of bytes in the hash, not just to each byte. This would be useful where the target device cannot read every byte of program memory and the hash value has to be padded.

3.13.  Version 2.20

In-built messages The compiler warnings and error messages, which are contained in a separate file, have now also been built into Hexmate and will be used when a message description file has not been specified.

New byte ordering option for Hexmate hash calculations A new revword=n suboption to the compiler driver's -mchecksum option requests Hexmate to read bytes in reverse order within n-byte words in the hex file for the purposes of calculating a hash, such as a checksum or CRC. This allows Hexmate to produce a result that will match that produced by devices that use the Scanner module to stream data to the CRC module in order to produce a CRC at runtime. Currently only word widths of 2 are supported. (Using the suboption but specifying n as 0 will also disable this reverse feature). If you are using Hexmate directly, use the corresponding rn suboption in Hexmate's -CK option.

SHA-ZAM The SHA256 algorithm has been added to Hexmate's suite of hash algorithms. It is accessible as algorithm #10 from Hexmate's -CK option, or from the compiler's -mchecksum option.

Type 3 HEX records Hexmate now processes type 3 records in HEX files, which might be produced when writing bootloaders for 8-bit AVR devices. These records are output verbatim in the final HEX file. A warning will be produced if more than one type 3 record is encountered and the record data is not consistent.

4.4.     Version 2.41

Generation of Hex files (XC8-2885) The generation of Hex files from linked object files is now performed by functionality that is part of Hexmate. This duty was previously performed by the Objtohex utility, which is no longer shipped with the compiler. This change should not impact any projects, as the Objtohex utility was only ever indirectly called by the compiler driver.

4.8.     Version 2.35

Hexmate II The Hexmate application shipped with the compiler is now built from a new code base. In addition, parts of the Hexmate code have been incorporated into the compiler drivers directly, meaning that the compiler no longer needs to execute the Hexmate application when building projects. If you run the compiler in verbose mode, a Hexmate application command line, like that shown here:

/Applications/microchip/xc8/v2.32/pic/bin/hexmate @/tmp/hexmate_xcNZD0QKH.cmd [ -W-3 --edf=/Applications/microchip/xc8/v2.32/pic/dat/en_msgs.txt main.hex -E1 -Omain.hex -logfile=main.hxl -addressing=1 -fill=0xFFFF@0x800:0xFFF -ck=800-FFF@20+0w1g2p0r0s0.1 -break=300000 ]

will no longer be seen. The stand-alone Hexmate application is still shipped with the compiler and if required can be run explicitly as it had in the past, allowing full access to its options or its use without a compiler. The log file output, when requested, shows more complete information relating to the tasks Hexmate performed.

5.   Fixed Issues

The following are corrections that have been made to Hexmate. The version number in the subheadings indicates the first version of Hexmate to contain fixes for the issues that follow. The bracketed label(s) in the title are that issue's identification in the tracking database. These may be useful if you need to contact support.

5.1.     Version 3.00

Record length limitation (HEXMATE-2) The -format option only permitted a maximum record length of 128 bytes. This has been increased to 255 bytes, which is the maximum permitted by Intel HEX files.

Silent failure on data conflict (HEXMATE-1) Hexmate exited with failure when processing an input HEX file that contained conflicting data, but failed to report such conflicts in the form of an error message. A message is now emitted in such situations.
